AgentFS architecture

How local files become safe source-of-record edits

AgentFS is a local-first Rust system. Editors, agents, the CLI, the desktop app, and platform filesystem adapters all converge on one daemon-owned execution boundary. The daemon serializes work, uses afs-core for sync decisions, persists facts in afs-store, and calls source-specific connectors through afs-connector. Notion is the first implemented source.

State model

What Owns Truth

The architecture stays comprehensible because each layer owns one kind of truth. The daemon is stateful, but the correctness logic lives in deterministic core APIs.

Three-tree sync model

The store keeps durable facts for remote, local, and synced state. Core compares them to classify the next action. That prevents ambiguous direction choices.

Remote tree

Last known source state from enumeration, scheduled pull, or relay later. Entity identity is RemoteId, not path.

Local tree

Current local projected state observed from files, virtual mutations, and dirty-state checks.

Synced tree

The merge base captured by the last reconciled shadow. Push and pull compare against this base.

Decision

sync.rs:10 returns pull, push, conflict, automerge, noop, or delete projection.

Trust and data boundaries

Content

Lives on disk in mount files, shadows, media/content cache, and in the source API. Direct mode has no AgentFS server in the content path.

Secrets

OAuth/PAT bytes are not serialized into mount files or JSON reports. SQLite stores connection metadata plus secret references.

Remote writes

Only approved push execution calls connector apply. Timed-out daemon push does not fall back to CLI to avoid duplicate mutation.

Recovery

Journals include preimages and apply effects so undo and failure recovery can reason from durable facts.
