Live Mode: How It Works and How to Debug It

This page explains Live Mode in plain English. It also explains what went wrong in this session, what was fixed, and what to check when Live Mode feels slow or stuck.

Main idea

Live Mode is a background helper. It can push safe local edits to Notion and pull clean Notion edits back to your file. It does not push every save blindly.

The Three Copies

Live Mode is easier to understand if you think about three copies of one page.

Local file

The file you edit

This is your page.md file in the mounted Locality folder.

→
Saved copy

Locality's last safe copy

Locality uses this to know what changed. In the code this is often called the shadow.

→
Remote page

The Notion page

This is the source of truth outside your computer.

Live Mode compares these copies. If only your local file changed, it may auto-push. If only Notion changed, it may auto-pull. If both changed, it pauses or creates conflict markers.

What Live Mode Does

Local to Notion

Auto-push safe edits

When you edit a file, the daemon marks it dirty. If the push plan is safe, Live Mode can push it to Notion.

Notion to local

Auto-pull clean edits

When Notion changes and the local file is clean, Live Mode can update the local file.

Safety

Pause when risky

Live Mode pauses for conflicts, large or destructive plans, unsupported edits, remote drift, or failed pushes.

Important: Live Mode is not instant. It uses queues, rate limits, and safety checks. A short delay is normal. A long delay usually means the daemon is not running, the file is paused, or the page is not being checked often enough.

Local Edit Flow

1

You save page.md

The OS or virtual filesystem sends a write event to Locality.

→
2

Locality marks it dirty

Status should show pending_local_changes or similar.

→
3

Auto-save may push

If Live Mode is enabled and the plan is safe, the daemon queues an auto-push.

What can stop it? What it means What to run
Live Mode is off for the file No file enrollment exists, or it is disabled. loc live-mode status <file> --json
The plan needs review The edit may be unsafe, unsupported, or destructive. loc diff <file>
Remote changed too Notion changed while local edits are waiting. loc status <file> then loc inspect <file>
The daemon is not running No background loop is processing jobs. loc daemon status

Remote Edit Flow

1

Notion changes

Locality sees a new remote version during a cheap remote check.

→
2

Locality checks the local file

If the local file is clean, it can fast-forward. If local work exists, it pauses or merges.

→
3

The file is updated

The daemon hydrates the page through the normal queue and refreshes the visible file if it is safe.

In the code: the cheap remote check is called freshness. The update job is called hydration. A Live Mode remote update uses LiveModeRemoteFastForward.

Common Problems

Paused

paused_remote_changed

Notion changed while Locality thought local work needed care. Run status and diff before pushing.

Blocked

blocked

The auto-save plan was not safe enough. You need to review it manually.

Slow remote updates

Freshness is not hot

The page may be checked only on the slower background path. File Live Mode should make this page active.

Missing child pages

Children were not listed yet

Workspace mounts do not scan all Notion on every tick. Child pages can appear after a child refresh or directory browse.

Notion lag

Remote version changed, body not ready

Notion can report a new edit time before all page blocks are ready. Locality retries instead of trusting a false clean state.

Wrong file path

Path did not resolve to a known page

Use the real mounted file. For Notion pages, the body file is usually page.md inside a page directory.

Debug Checklist

Use this order. It avoids guessing.

  1. Check that Live Mode is on for this file. 
    loc live-mode status "/path/to/page.md" --json
  2. Check what Locality thinks about the file. 
    loc status "/path/to/page.md" --json

    Look for sync_state, hydration, issues, and remote version fields.

  3. Check the daemon. 
    loc daemon status

    Look at active jobs, pending hydrations, and pending freshness jobs.

  4. If local changes are waiting, review the plan. 
    loc diff "/path/to/page.md"
  5. If remote changed, inspect the exact difference. 
    loc inspect "/path/to/page.md" --json
  6. Only push manually when you mean to. 
    loc push "/path/to/page.md" -y

    Agents should not run push after every edit when Live Mode is on.

What Happened In This Session

Start

We studied Live Mode first

We read the Live Mode behavior, CLI code, daemon runtime code, tests, and docs. The important parts were auto-save enrollments, freshness jobs, hydration jobs, and status output.

Blocker 1

Windows path test failed

The test compared paths with different separators: Windows returned Roadmap\page.md, but the test expected Roadmap/page.md. The fix was to compare normalized paths instead of a mixed string.

Blocker 2

A mounted directory had only page.md at first

The page had child pages in Notion, but they were not in the local directory yet. This can happen because workspace virtual mounts do not scan all children immediately. Child pages appear after child refresh work runs or after the directory is browsed.

Blocker 3

Local edits did not auto-push

The file had Live Mode enabled, but file-level Live Mode only created an auto-save enrollment. Some daemon freshness behavior still only looked at mount-level Live Mode. So the page was not treated as an active Live Mode target.

Blocker 4

Remote edits came back slowly

Because the file-level enrollment was not used by the faster freshness path, remote checks could wait for slower background timing.

Blocker 5

Clean remote changes paused auto-save

The daemon paused the file when it saw a remote version change, even if the local file was clean and safe to update. That made the file look stuck in paused_remote_changed.

Fix

PR #75 fixed the file-level path

The fix made enabled file Live Mode enrollments active freshness targets. It also kept clean enrolled files active on remote drift and queued LiveModeRemoteFastForward. Dirty or conflicted files still pause.

Result

Tests and CI passed

Local tests passed, then CI passed on Linux, macOS, Windows, OAuth service, Local e2e behavior, and Linux FUSE e2e.

Quick Meaning Of States

State Simple meaning Usual next step
all_synced Locality knows no local or remote change waiting. No action.
checking_freshness Locality is checking if Notion changed. Wait a little, then check status again.
pending_local_changes Your file changed and has not been pushed yet. Use loc diff. Live Mode may push if safe.
remote_update_available Notion changed and your local file looks clean. Live Mode may fast-forward. Use loc inspect if unsure.
review_needed Locality needs a person or agent to decide. Run loc diff and loc inspect.
conflicted There are conflict markers or a conflict state. Edit the file to resolve it, then review and push.

Good Rules For Agents

  • Edit the mounted Markdown file directly.
  • Use loc status and loc diff to understand state.
  • Do not run loc push after every edit if Live Mode is on.
  • Run loc push only when the user asks, or when Live Mode paused and the plan is safe.
  • If both local and remote changed, do not force it. Inspect first.
